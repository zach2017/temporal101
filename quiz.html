What is DocPulse?

DocPulse is a Spring Boot 3.5 application that demonstrates two patterns for asynchronous document processing — a lightweight CompletableFuture approach and a durable Temporal Worker pipeline. This tutorial covers the Temporal path end-to-end.

⚡

CompletableFuture Mode

Runs in-process on Spring's @Async thread pool. Simple, zero dependencies. If the server restarts mid-processing, the job is lost.

Fast startup Not durable
🔄

Temporal Worker Mode

Runs via the Temporal orchestration platform. Every step is persisted. Server restarts, crashes, and retries are handled automatically with full event history.

Durable Auto-retry Observable

Core Temporal Concepts

Understanding these five primitives is all you need to build production Temporal applications.

🔁

Workflow

A Workflow is a durable, fault-tolerant orchestrator. It defines the sequence of Activities to execute and the logic between them. Temporal persists its complete event history so it can resume after any failure.

Key rule: Workflow code must be deterministic — given the same event history it always produces the same execution path. Never call System.currentTimeMillis() or Thread.sleep() inside a Workflow.

docs.temporal.io → Workflow concepts ↗
Java SDK → Workflows ↗ Python SDK → Workflows ↗
⚙️

Activity

An Activity is the unit of work — I/O, HTTP calls, database writes, OCR, NLP inference. Activities ARE allowed to be non-deterministic. They run on a Worker's thread pool and report heartbeats so Temporal knows they are alive.

Activities have their own retry policy (max attempts, backoff) configured via ActivityOptions.

docs.temporal.io → Activity concepts ↗
Java SDK → Activities ↗ Python SDK → Activities ↗
👷

Worker

A Worker is a long-running process that polls a Task Queue for Workflow and Activity tasks, executes the matching implementation, and reports the result back to Temporal.

In DocPulse, DocumentWorkerRegistrar creates a WorkerFactory, registers the three Activity impls and the WorkflowImpl, then starts polling DOCUMENT_PROCESSING_TASK_QUEUE.

docs.temporal.io → Worker concepts ↗
Java SDK → Workers ↗ Python SDK → Workers ↗
📋

Task Queue

A Task Queue is a named channel between the Temporal server and your Workers. When a Workflow schedules an Activity, Temporal places a task on the queue. A Worker polling that queue picks it up and executes it.

DocPulse uses one queue: DOCUMENT_PROCESSING_TASK_QUEUE. You can scale horizontally by running more Worker processes on the same queue.

docs.temporal.io → Task Queue concepts ↗
Java SDK → Task Queues ↗ Python SDK → Task Queues ↗
📡

Signals & Queries

@QueryMethod — a read-only, strongly-consistent view into the Workflow's in-memory state. The DocPulse frontend calls GET /api/worker/status/{id} which triggers getStatus() — no database needed.

@SignalMethod — an asynchronous message sent to a running Workflow. DocPulse uses requestCancellation(reason) to gracefully stop processing between Activity steps.

⚠️ Determinism Rules — The Most Important Concept

❌ Never in Workflow code:

  • System.currentTimeMillis()
  • Thread.sleep()
  • new Random() / UUID.randomUUID()
  • Direct DB/HTTP/File I/O
  • System.getenv()

✅ Always use instead:

  • Workflow.currentTimeMillis()
  • Workflow.sleep(Duration)
  • Workflow.newRandom()
  • Put I/O inside Activities
  • Workflow.getInfo() for metadata

private final SecurityScanActivity scanActivity = Workflow.newActivityStub( SecurityScanActivity.class, ActivityOptions.newBuilder() // Total wall-clock budget including all retries .setScheduleToCloseTimeout(Duration.ofMinutes(3)) // Max time for a single attempt .setStartToCloseTimeout(Duration.ofMinutes(2)) // If no heartbeat in 10 s → attempt is considered failed .setHeartbeatTimeout(Duration.ofSeconds(10)) .setRetryOptions(RetryOptions.newBuilder() .setMaximumAttempts(3) .setInitialInterval(Duration.ofSeconds(1)) .setBackoffCoefficient(2.0) .build()) .build() );

Heartbeats in Activities

Any activity longer than heartbeatTimeout must call ctx.heartbeat() regularly. DocPulse does this every 200 ms inside a helper:

private static void simulateWork( ActivityExecutionContext ctx, long millis, String detail) { long remaining = millis; while (remaining > 0) { long chunk = Math.min(200, remaining); try { Thread.sleep(chunk); } catch (InterruptedException e) { Thread.currentThread().interrupt(); return; } ctx.heartbeat(detail); // ← tells Temporal we're alive remaining -= chunk; } }

Workers Deep-Dive

How DocPulse's workers are annotated, registered, and started — and how to add new ones.

Auto-Discovery Annotations

The temporal-spring-boot-autoconfigure starter scans the package defined in workers-auto-discovery.packages for these annotations:

// Workflow implementation @WorkflowImpl(taskQueues = "DOCUMENT_PROCESSING_TASK_QUEUE") public class DocumentProcessingWorkflowImpl implements DocumentProcessingWorkflow { … } // Activity implementation (one per stage) @Component @ActivityImpl(taskQueues = "DOCUMENT_PROCESSING_TASK_QUEUE") public class SecurityScanActivityImpl implements SecurityScanActivity { … }
docs → Core Application (Java) ↗ docs → Core Application (Python) ↗

Explicit Worker Registration

DocumentWorkerRegistrar also does manual registration as a fallback (and shows the pattern for more complex setups):

WorkerFactory factory = WorkerFactory.newInstance(workflowClient); Worker worker = factory.newWorker( DocumentWorker.TASK_QUEUE, WorkerOptions.newBuilder() .setMaxConcurrentWorkflowTaskExecutionSize(10) .setMaxConcurrentActivityExecutionSize(20) .build()); worker.registerWorkflowImplementationTypes( DocumentProcessingWorkflowImpl.class); worker.registerActivitiesImplementations( scanActivity, extractActivity, nlpActivity); factory.start();

How to Add a New Activity or Worker

Step 1 — Define the Activity interface

Create a new interface annotated with @ActivityInterface. Define methods annotated with @ActivityMethod. Add a nested result POJO.

Step 2 — Implement it

Annotate the impl class with @Component and @ActivityImpl(taskQueues = "DOCUMENT_PROCESSING_TASK_QUEUE"). Add ctx.heartbeat() calls for long steps.

Step 3 — Add a stub in the Workflow

In DocumentProcessingWorkflowImpl, create a new Workflow.newActivityStub() field with ActivityOptions.

Step 4 — Call it in processDocument()

Add the call to your stub after the existing stages. Update updateStatus() with the new PipelineStage value before calling it.

Step 5 — Register in WorkerRegistrar (if not using auto-discovery)

Inject the new Activity impl bean and add it to worker.registerActivitiesImplementations(...).

Step 6 — Restart and verify in the Temporal Web UI

Open http://localhost:8233, find your workflow run, and inspect the event history to confirm the new Activity task appears.

🚀 Scaling Workers

To scale horizontally, simply run more instances of the Spring Boot app. Each instance starts its own Worker polling the same task queue. Temporal distributes tasks round-robin across all available Workers automatically. No leader election or coordination code required.

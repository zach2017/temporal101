How It Works

A deep dive into the Temporal SDK code, worker architecture, and platform capabilities.

1 Workflows & Activities

Temporal applications are built around two primitives: Workflows (durable functions that orchestrate logic) and Activities (units of work that can fail and be retried). Workflows are deterministic and their execution state is persisted by the Temporal Server. Activities are where your side effects live — API calls, database writes, sending emails.

Workflow

A durable, fault-tolerant function. Its full execution history is stored by Temporal. If a worker crashes, the workflow is seamlessly resumed on another worker via event replay.

Activity

A single unit of work within a workflow. Activities can be retried automatically on failure with configurable retry policies, timeouts, and heartbeat intervals.

Task Queue

A named queue that routes workflow and activity tasks to the correct set of workers. Multiple workers can poll the same queue for load balancing.

2 Java Worker — SDK Code

The Java worker uses io.temporal:temporal-sdk:1.32.0. Workflows are defined as interfaces annotated with @WorkflowInterface, and activities with @ActivityInterface.

Java@WorkflowInterface
public interface JavaHelloWorkflow {
    @WorkflowMethod
    String sayHello(String name);
}

public class JavaHelloWorkflowImpl implements JavaHelloWorkflow {
    // Activity stub with retry policy
    private final GreetingActivities activities =
        Workflow.newActivityStub(GreetingActivities.class,
            ActivityOptions.newBuilder()
                .setStartToCloseTimeout(Duration.ofSeconds(10))
                .setRetryOptions(RetryOptions.newBuilder()
                    .setMaximumAttempts(3).build())
                .build());

    @Override
    public String sayHello(String name) {
        return activities.composeGreeting(name);
    }
}
Java@ActivityInterface
public interface GreetingActivities {
    String composeGreeting(String name);
}

public class GreetingActivitiesImpl implements GreetingActivities {
    @Override
    public String composeGreeting(String name) {
        return "Hello " + name + " from Java Worker! (JDK 21, Temporal SDK 1.32.0)";
    }
}

The worker process registers these implementations and polls the java-hello-queue task queue:

JavaWorkerFactory factory = WorkerFactory.newInstance(client);
Worker worker = factory.newWorker("java-hello-queue");
worker.registerWorkflowImplementationTypes(JavaHelloWorkflowImpl.class);
worker.registerActivitiesImplementations(new GreetingActivitiesImpl());
factory.start(); // begins long-polling the task queue

3 Python Worker — SDK Code

The Python worker uses the temporalio package. Workflows use the @workflow.defn decorator and activities use @activity.defn.

Python@activity.defn
async def compose_greeting(name: str) -> str:
    return f"Hello {name} from Python Worker! (temporalio SDK)"


@workflow.defn
class PythonHelloWorkflow:
    @workflow.run
    async def run(self, name: str) -> str:
        return await workflow.execute_activity(
            compose_greeting,
            name,
            start_to_close_timeout=timedelta(seconds=10),
            retry_policy=RetryPolicy(maximum_attempts=3),
        )
Python# Start the worker, polling python-hello-queue
worker = Worker(
    client,
    task_queue="python-hello-queue",
    workflows=[PythonHelloWorkflow],
    activities=[compose_greeting],
)
await worker.run()

4 Spring Boot Client

The Spring Boot app acts as the Temporal Client. It creates workflow stubs and starts executions. For the Java worker, it uses a typed stub (shared interface). For the Python worker, it uses an untyped stub (just the workflow type name string).

Java// Typed stub for Java worker
JavaHelloWorkflow javaWf = workflowClient.newWorkflowStub(
    JavaHelloWorkflow.class,
    WorkflowOptions.newBuilder()
        .setTaskQueue("java-hello-queue")
        .build());
String result = javaWf.sayHello(name); // blocks until complete

// Untyped stub for Python worker (cross-language)
WorkflowStub pythonWf = workflowClient.newUntypedWorkflowStub(
    "PythonHelloWorkflow",
    WorkflowOptions.newBuilder()
        .setTaskQueue("python-hello-queue")
        .build());
pythonWf.start(name);
String result = pythonWf.getResult(String.class);

5 Resiliency & Fault Tolerance

Temporal provides extraordinary resiliency out of the box. Here is what happens when things go wrong:

  • Worker crashes mid-workflow: Temporal persists every workflow step as an event in its history. When a new worker picks up the task, it replays events to reconstruct state and continues from exactly where it left off. No work is lost.
  • Activity fails (network timeout, API error): The built-in retry policy kicks in automatically. You configure setMaximumAttempts, setBackoffCoefficient, and setInitialInterval. Temporal handles exponential backoff for you.
  • Temporal Server itself restarts: All workflow state is persisted in PostgreSQL. When the server comes back, all workflows resume from their last committed state.
  • Network partition between worker and server: Workers use long-polling with automatic reconnection. Workflows remain safe in the server's database until the worker reconnects.
  • Duplicate workflow starts: Workflow IDs provide natural idempotency. Starting a workflow with the same ID will either reject the duplicate or return the existing execution.

Key insight: Traditional code loses state on crashes and requires manual retry logic, circuit breakers, and state machines. Temporal eliminates all of that — your business logic reads like simple, linear code, while Temporal handles all the failure modes transparently.

6 Scalability — Adding More Workers

Scaling Temporal is straightforward: add more workers. Multiple worker processes can poll the same task queue, and Temporal distributes tasks among them automatically.

Docker# Scale to 5 Java workers with one command:
docker compose up --scale java-worker=5

# Scale to 3 Python workers:
docker compose up --scale python-worker=3
  • Horizontal scaling: Each worker is a stateless process. Add or remove workers at any time without affecting running workflows.
  • Task routing: Task queues decouple workflow scheduling from execution. You can have specialized queues for different workload types (e.g., cpu-intensive-queue, io-bound-queue).
  • Multi-language polyglot: As this demo shows, Java and Python workers coexist seamlessly. You could add Go, TypeScript, or .NET workers on separate task queues.
  • Rate limiting: Temporal supports server-side rate limiting per task queue to prevent worker overload during traffic spikes.
  • Worker auto-tuning (SDK 1.32+): The Java SDK supports PollerBehaviorAutoscaling which automatically adjusts the number of concurrent polls based on load.

7 Audit Trails & Observability

Every workflow execution in Temporal has a complete, immutable Event History. This is not just logging — it is the authoritative record of everything that happened.

Event History

Every state transition (WorkflowStarted, ActivityScheduled, ActivityCompleted, WorkflowCompleted) is recorded as a numbered event with timestamps and payloads.

Temporal Web UI

Browse workflow executions at localhost:8080. Inspect the full event history, input/output payloads, retry attempts, and timing for every workflow.

Searchable via Visibility

Use SQL-like queries to find workflows by ID, type, status, or custom search attributes. Essential for compliance and debugging at scale.

You can query workflow history programmatically or view it in the Temporal UI. Each workflow run in this demo generates events like:

Temporal.io Deep Dive

Document Management with Temporal

Build a fault-tolerant document processing pipeline using Temporal workflows, LocalStack S3, Elasticsearch, and Tesseract OCR — with Docker Compose.

How Temporal works

Temporal is a durable execution engine. You write workflows as ordinary code — loops, conditionals, try/catch — and Temporal makes that code survive crashes, restarts, and network failures by recording every step in a persistent event history.

The three core primitives

W

Workflows

Orchestrators. Deterministic code that sequences activities, handles errors, and maintains state. Workflows can run for seconds or years.

A

Activities

Side effects. API calls, S3 uploads, database writes, OCR processing — anything non-deterministic goes here. Independently retryable.

Q

Task queues

Routing layer. Workers poll task queues for work. Multiple workers can poll the same queue for horizontal scaling.

The event history model

Every decision your workflow makes — every activity call, every timer, every signal — is appended as an event to a durable log stored in the database. This is the key to fault tolerance:

1

Client starts a workflow

Your app sends execute_workflow(ProcessDocument, "report.pdf") to the Temporal Server. The server creates an event history with WorkflowExecutionStarted.

2

Worker picks up the workflow task

A worker polling the task queue receives the task. It runs your workflow code up to the first execute_activity() call — say, detect_file_type.

3

Activity executes as a separate task

The server creates an ActivityTaskScheduled event. A worker picks it up, runs the file detection, and reports the result. The server records ActivityTaskCompleted with the return value.

4

Workflow continues to the next step

The worker receives the activity result and advances the workflow to the next execute_activity() — storing in S3. The cycle repeats for each step.

5

If a worker crashes...

Another worker picks up the task and replays the event history. Completed activities are skipped (their results are already recorded). The workflow resumes exactly where it left off.

The determinism rule: Workflow code must produce the same decisions given the same event history. No random(), no datetime.now(), no I/O. All side effects go in Activities.

Why this matters for document processing

A PDF processing pipeline touches many external systems — S3, Elasticsearch, Tesseract, the filesystem. Any of them can fail at any time. Without Temporal, you'd need to build retry logic, dead-letter queues, state checkpointing, and idempotency guards for every step. With Temporal, you write text = await extract_text_from_pdf(path) and the platform handles the rest.

Scaling & failure recovery

How Temporal scales this pipeline

Temporal scales at three levels: the server, the task queues, and your workers. Each level is independent.

1

Worker-level scaling

Run more python worker.py instances. Each polls the same task queue. Double the workers → double the throughput. No config changes needed.

2

Queue-level scaling

Route different activities to different queues. Put OCR on document-ocr queue with GPU workers. Put storage on document-storage queue near the S3 cluster.

3

Server-level scaling

Temporal's server is sharded. Add more History Service instances → more concurrent workflows. Add more Matching Service instances → more task queue throughput.

Scaling strategy for this pipeline

Failure recovery scenarios

Here's what happens when things go wrong — and why your code doesn't need to handle any of it:

Scenario: Worker crashes during OCR
The OCR activity sends heartbeats every 60 seconds. When a heartbeat is missed, Temporal marks the activity as timed out, puts it back on the task queue, and another worker picks it up. The workflow is unaware a crash happened — it just awaits the result.
Scenario: S3 (LocalStack) is temporarily unreachable
The store_document_s3 activity fails. Temporal retries with exponential backoff: 2s → 4s → 8s → 16s → 32s, up to 5 attempts. If S3 comes back within that window, the activity succeeds. The workflow never sees the transient failure.
Scenario: Elasticsearch is down for 10 minutes
Text storage to S3 and filesystem complete normally (they're independent activities). Only the ES activities retry. When ES recovers, the pending activities succeed and the workflow finishes. Partial failures don't block the rest of the pipeline.
Scenario: The entire Temporal server restarts
All event histories are persisted in PostgreSQL. When the server comes back, every in-flight workflow resumes from its last recorded event. Workers reconnect and continue polling. Zero data loss, zero manual intervention.

Activity implementations

Each activity is a standalone async function decorated with @activity.defn. They contain all the "dangerous" code — I/O, network calls, OCR — that Temporal makes retryable.

Storage: upload to S3

storage_activities.pyPYTHON
@activity.defn
async def store_document_s3(
    doc_id: str, file_path: str, content_type: str
) -> str:
    """Upload raw document to LocalStack S3."""
    s3 = boto3.client("s3",
        endpoint_url="http://localhost:4566")
    s3_key = f"documents/{doc_id}/{os.path.basename(file_path)}"

    with open(file_path, "rb") as f:
        s3.put_object(
            Bucket="docmgr-raw-documents",
            Key=s3_key,
            Body=f.read(),
            ContentType=content_type,
        )
    return s3_key

Text extraction: PDF

text_extraction_activities.pyPYTHON
@activity.defn
async def extract_text_from_pdf(file_path: str) -> dict:
    """Extract text with PyPDF2, fallback to pdfplumber."""
    import PyPDF2

    with open(file_path, "rb") as f:
        reader = PyPDF2.PdfReader(f)
        text_pages = []

        for i, page in enumerate(reader.pages):
            text_pages.append(page.extract_text() or "")
            # Heartbeat tells Temporal we're still alive
            activity.heartbeat(f"Page {i+1}/{len(reader.pages)}")

    return {
        "text": "\n\n".join(text_pages),
        "page_count": len(reader.pages),
    }

OCR: Tesseract on images

ocr_activities.pyPYTHON
@activity.defn
async def ocr_single_image(image_path: str) -> dict:
    """Extract text from an image using Tesseract OCR."""
    import pytesseract
    from PIL import Image, ImageFilter

    img = Image.open(image_path)
    img = img.convert("L")             # Grayscale
    img = img.filter(ImageFilter.SHARPEN) # Sharpen edges

    text = pytesseract.image_to_string(img, lang="eng")

    # Get confidence scores
    data = pytesseract.image_to_data(
        img, output_type=pytesseract.Output.DICT)
    confidences = [int(c) for c in data["conf"]
                   if str(c).strip() and int(c) > 0]

    return {
        "text": text.strip(),
        "confidence": sum(confidences) / len(confidences),
    }

Elasticsearch indexing

storage_activities.pyPYTHON
@activity.defn
async def store_text_elasticsearch(
    doc_id: str, text: str,
    filename: str, file_type: str, metadata: dict,
) -> str:
    """Index extracted text in ES for full-text search."""
    es = Elasticsearch("http://localhost:9200")

    result = es.index(
        index="document_text",
        id=doc_id,
        body={
            "doc_id": doc_id,
            "filename": filename,
            "text_content": text,
            "word_count": len(text.split()),
            **metadata,
        },
    )
    return result["_id"]

The workflow orchestrator

The DocumentProcessingWorkflow ties everything together. Notice how it reads like a normal Python script — but every await is a durable checkpoint. If the process crashes after step 3, Temporal replays steps 1-3 from history and continues from step 4.

document_workflow.py — core logicPYTHON
@workflow.defn
class DocumentProcessingWorkflow:

    @workflow.run
    async def run(self, file_path: str, doc_id: str) -> dict:

        # Step 1: Detect file type
        file_info = await workflow.execute_activity(
            detect_file_type, file_path,
            start_to_close_timeout=timedelta(seconds=30),
        )

        # Step 2: Store raw document (S3 + filesystem in PARALLEL)
        s3_task = workflow.execute_activity(
            store_document_s3, args=[doc_id, file_path, file_info["mime_type"]],
            start_to_close_timeout=timedelta(seconds=120),
        )
        fs_task = workflow.execute_activity(
            store_document_filesystem, args=[doc_id, file_path],
            start_to_close_timeout=timedelta(seconds=60),
        )
        s3_key = await s3_task   # Both run concurrently
        fs_path = await fs_task

        # Step 3: Route to correct extractor
        category = file_info["category"]

        if category == "pdf":
            text_result = await workflow.execute_activity(
                extract_text_from_pdf, file_path, ...)
            # Also extract images from the PDF
            images = await workflow.execute_activity(
                extract_images_from_pdf, args=[file_path, doc_id], ...)
            # OCR on the extracted images
            ocr = await workflow.execute_activity(
                ocr_batch_images, images["image_paths"], ...)

        elif category == "image":
            text_result = await workflow.execute_activity(
                ocr_single_image, file_path, ...)

        elif category == "docx":
            text_result = await workflow.execute_activity(
                extract_text_from_docx, file_path, ...)

        # Step 4: Store text everywhere (PARALLEL)
        await workflow.execute_activity(store_text_s3, ...)
        await workflow.execute_activity(store_text_filesystem, ...)
        await workflow.execute_activity(store_text_elasticsearch, ...)

        # Step 5: Index metadata
        await workflow.execute_activity(index_document_metadata, ...)

        return result

handle = client.get_workflow_handle("doc-123")

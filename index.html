What is Temporal?

Temporal is an open-source durable execution platform that lets you write complex application logic as simple procedural code. Instead of juggling message queues, retry logic, state machines, and cron schedulers, you write a Workflow — a function that can run for seconds, hours, or even years — and Temporal guarantees it will complete, even through process crashes, network failures, and server restarts.

Core idea: Your code writes like a normal function with try/catch, loops, and if-else. Temporal makes that function indestructible by recording every step in a durable event history.

Why do teams use Temporal?

Before Temporal, building reliable distributed systems meant hand-wiring retries, timeouts, state persistence, idempotency keys, dead-letter queues, and compensating transactions. A single order-processing pipeline might need dozens of infrastructure primitives glued together. Temporal replaces all of that with one programming model.

How Temporal works

Temporal has a client-server architecture with clear separation between the orchestration engine (Temporal Server) and your business logic (Workers).

The execution model

1

Client starts a Workflow

Your app sends a gRPC request to the Temporal Server saying "run this workflow with these inputs." The server assigns it a Workflow ID and creates an event history.

2

Server schedules a Workflow Task

The Matching Service puts a task onto a Task Queue. Think of it like a smart mailbox — tasks wait here until a Worker picks them up.

3

Worker picks up the task

Workers long-poll task queues. When a worker grabs a Workflow Task, it replays the event history to rebuild state, then executes the next step of your workflow code.

4

Activities execute side effects

When the workflow calls an Activity (API call, DB write, sending email), the server creates an Activity Task. A worker picks it up, executes the real work, and reports the result.

5

Events are durably recorded

Every decision, activity result, timer, and signal is appended to the workflow's Event History in the database. If the worker crashes mid-execution, another worker replays the history and continues exactly where it left off.

Key insight — Replay: Workflow code must be deterministic. Temporal doesn't persist variables — it persists events and replays your workflow code against them. Non-deterministic calls (random numbers, current time, I/O) must go inside Activities, not workflow code.

Features & Scaling

Core features

⚡

Durable Timers

Sleep for minutes or months inside a workflow. No cron, no external scheduler — just workflow.sleep("30d").

📡

Signals

Send data to a running workflow from outside. Example: user approves an order → signal the waiting workflow to continue.

🔍

Queries

Read workflow state without affecting execution. Check order status, progress percentage, or any internal variable.

🔁

Continue-As-New

Long-running workflows can "restart" with fresh history to avoid unbounded event logs while preserving state.

🧒

Child Workflows

Compose complex workflows from simpler ones. A parent workflow can spawn children that run independently.

📋

Visibility & Search

Query running/completed workflows by custom attributes. Full observability via the Temporal Web UI.

🔒

Versioning

Safely deploy new workflow logic while old executions finish on the previous version — no breaking changes.

📅

Schedules

Built-in cron-like scheduling. Run workflows on a cadence without external schedulers.

How Temporal scales

Temporal was designed for massive scale from the start. Here's how each layer scales independently:

ComponentScaling StrategyNotes
Frontend ServiceHorizontal — add more instancesStateless gRPC gateway, load-balanced
Matching ServiceHorizontal + sharded by Task QueueEach task queue partition handles its own workers
History ServiceHorizontal + sharded by Workflow IDEach shard owns a subset of workflow histories
WorkersHorizontal — add more worker processesWorkers are stateless; add as many as needed

Production numbers: Temporal has been battle-tested at companies handling millions of concurrent workflows. The server can process tens of thousands of workflow transitions per second per shard. Workers scale linearly — double the workers, double the throughput.

5. Implement the Workflow

GreetingWorkflowImpl.javaJAVA
package com.example.temporal;

import io.temporal.activity.ActivityOptions;
import io.temporal.workflow.Workflow;
import java.time.Duration;

/**
 * Workflow implementation. This code is
 * DETERMINISTIC — no I/O, no random(), no
 * System.currentTimeMillis(). Use Activities
 * for all side effects.
 */
public class GreetingWorkflowImpl
       implements GreetingWorkflow {

    // Create an Activity stub with options
    private final GreetingActivities activities =
        Workflow.newActivityStub(
            GreetingActivities.class,
            ActivityOptions.newBuilder()
                .setStartToCloseTimeout(
                    Duration.ofSeconds(10))
                .build()
        );

    @Override
    public String greet(String name) {
        // This looks like normal code, but Temporal
        // records each Activity call as an event.
        // If the worker crashes here, another worker
        // replays the history and skips completed steps.
        String result = activities
            .composeGreeting("Hello", name);

        // You can call multiple activities,
        // use loops, if-else, try-catch — all normal!
        return result;
    }
}

6. Create the Worker

GreetingWorker.javaJAVA
package com.example.temporal;

import io.temporal.client.WorkflowClient;
import io.temporal.serviceclient.WorkflowServiceStubs;
import io.temporal.worker.Worker;
import io.temporal.worker.WorkerFactory;

/**
 * The Worker process: connects to the Temporal
 * Server and polls a Task Queue for work.
 */
public class GreetingWorker {

    public static final String TASK_QUEUE =
        "greeting-tasks";

    public static void main(String[] args) {
        // 1. Connect to Temporal Server
        WorkflowServiceStubs service =
            WorkflowServiceStubs.newLocalServiceStubs();
        WorkflowClient client =
            WorkflowClient.newInstance(service);

        // 2. Create a Worker Factory
        WorkerFactory factory =
            WorkerFactory.newInstance(client);

        // 3. Create a Worker that polls TASK_QUEUE
        Worker worker =
            factory.newWorker(TASK_QUEUE);

        // 4. Register Workflow & Activity impls
        worker.registerWorkflowImplementationTypes(
            GreetingWorkflowImpl.class);
        worker.registerActivitiesImplementations(
            new GreetingActivitiesImpl());

        // 5. Start polling (blocks this thread)
        factory.start();
        System.out.println(
            "Worker started on: " + TASK_QUEUE);
    }
}

7. Start a Workflow (Client)

GreetingStarter.javaJAVA
package com.example.temporal;

import io.temporal.client.WorkflowClient;
import io.temporal.client.WorkflowOptions;
import io.temporal.serviceclient.WorkflowServiceStubs;

public class GreetingStarter {

    public static void main(String[] args) {
        // Connect to Temporal Server
        WorkflowServiceStubs service =
            WorkflowServiceStubs.newLocalServiceStubs();
        WorkflowClient client =
            WorkflowClient.newInstance(service);

        // Create a typed Workflow stub
        GreetingWorkflow workflow =
            client.newWorkflowStub(
                GreetingWorkflow.class,
                WorkflowOptions.newBuilder()
                    .setWorkflowId("greeting-1")
                    .setTaskQueue(
                        GreetingWorker.TASK_QUEUE)
                    .build()
            );

        // Execute the Workflow (blocking call)
        String result = workflow.greet("World");
        System.out.println(
            "Workflow result: " + result);
        // Output: Workflow result: Hello World!

        System.exit(0);
    }
}

from temporalio import workflow

# Import Activities using their module path.
# This avoids importing non-deterministic code
# directly into the workflow sandbox.
with workflow.unsafe.imports_passed_through():
    from activities import compose_greeting


@workflow.defn
class GreetingWorkflow:

    Workflow definition. This code must be

    datetime.now(). Use Activities for side effects.
    """

    @workflow.run
    async def run(self, name: str) -> str:
        # Call the activity with retry options
        result = await workflow.execute_activity(
            compose_greeting,
            args=["Hello", name],
            start_to_close_timeout=timedelta(seconds=10),
        )

        # You can use loops, if-else, try-except

4. Create the Worker

worker.pyPYTHON
import asyncio
from temporalio.client import Client
from temporalio.worker import Worker

from activities import compose_greeting
from workflows import GreetingWorkflow

TASK_QUEUE = "greeting-tasks-python"


async def main():
    # 1. Connect to the Temporal Server
    client = await Client.connect(
        "localhost:7233"
    )

    # 2. Create and run the Worker
    worker = Worker(
        client,
        task_queue=TASK_QUEUE,
        workflows=[GreetingWorkflow],
        activities=[compose_greeting],
    )

    print(f"Worker started on: {TASK_QUEUE}")

    # 3. Run forever (polls for tasks)
    await worker.run()


if __name__ == "__main__":
    asyncio.run(main())

Bonus: Advanced Python example (Signals + Queries)

advanced_workflow.pyPYTHON
from datetime import timedelta
from temporalio import workflow

with workflow.unsafe.imports_passed_through():
    from activities import compose_greeting


@workflow.defn
class ApprovalWorkflow:
    """
    A workflow that waits for human approval.
    Demonstrates Signals, Queries, and Timers.
    """

    def __init__(self):
        self.approved = False
        self.status = "pending"

    @workflow.run
    async def run(self, name: str) -> str:

        # Wait for approval OR timeout (24 hours)
        try:
            await workflow.wait_condition(
                lambda: self.approved,
                timeout=timedelta(hours=24),
            )
            self.status = "approved"
        except asyncio.TimeoutError:
            self.status = "timed_out"
            return "Approval timed out for " + name

        # Approval received — run the activity
        result = await workflow.execute_activity(
            compose_greeting,
            args=["Welcome", name],
            start_to_close_timeout=timedelta(seconds=10),
        )
        self.status = "completed"
        return result

    @workflow.signal
    async def approve(self):
        """External signal to approve the request."""
        self.approved = True

    @workflow.query
    def get_status(self) -> str:
        """Query current status without side effects."""
        return self.status

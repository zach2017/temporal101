# File Shuttle

Memory-efficient streaming file transfer between **S3**, **NFS**, **HTTP/URL**, and **local filesystems**.  
Java 21 CLI app · Maven · Docker Compose · AWS SDK v2

---

## Architecture Overview

```
                          ┌─────────────────────────────────┐
                          │     FileShuttleApp (CLI)         │
                          │  Parses args → TransferRequest   │
                          └──────────────┬──────────────────┘
                                         │
                          ┌──────────────▼──────────────────┐
                          │    TransferOrchestrator          │
                          │  Pipes InputStream→OutputStream  │
                          │  Collects speed + memory metrics │
                          └──────┬───────────────┬──────────┘
                                 │               │
                    ┌────────────▼───┐   ┌───────▼────────────┐
                    │ INPUT Provider │   │ OUTPUT Provider     │
                    │ .openInput()   │   │ .openOutput()       │
                    └────────────────┘   └────────────────────┘
                           │                       │
              ┌────────────┼────────┐    ┌─────────┼────────┐
              │     │       │       │    │    │     │        │
            Local  S3     NFS    URL   S3   NFS   URL    Local
```

### Data Flow — Zero Full-File Buffering

```
Source ──read──▶ [64KB buffer] ──write──▶ Destination

S3 input:    HTTP chunked stream (lazy, pulled on read)
S3 output:   8MB part buffer → uploadPart → reuse buffer
URL input:   HTTP response InputStream (lazy)
URL output:  PipedOutputStream → PipedInputStream → async HTTP body
Local/NFS:   BufferedInputStream/BufferedOutputStream (64KB)
```

## Shared Model Classes (the "contract")

### LocationType
Enum with 4 values: `LOCAL`, `S3`, `NFS`, `URL`.  
Includes `fromString()` with aliases (`"http"` → URL, `"file"` → LOCAL, etc.)  
Used everywhere: CLI parsing, provider routing, metrics labeling.

### TransferRequest  
Immutable Java `record` — the single object passed through every layer:
- `fileName` — logical name for logging
- `inputType` / `inputLocation` — source backend + address
- `outputType` / `outputLocation` — destination backend + address
- `options` — tuning map (bufferSize, multipartPartSize, httpMethod)
- Helper methods: `bufferSize()`, `multipartPartSize()`, `httpMethod()`

### TransferResult
Immutable `record` returned by the orchestrator:
- `bytesTransferred`, `startTime`, `endTime`, `peakMemoryBytes`
- Derived: `duration()`, `throughputMBps()`, `summary()`
- Factories: `TransferResult.success(...)`, `TransferResult.failure(...)`

## Provider Details

### StorageProvider Interface
```java
InputStream  openInputStream(TransferRequest request)   // lazy read
OutputStream openOutputStream(TransferRequest request)  // incremental write
long         contentLength(TransferRequest request)      // for progress
String       name()                                      // for logging
```

### LocalFSProvider (LOCAL + NFS)
- **Read:** `BufferedInputStream` wrapping `Files.newInputStream` (NIO)
- **Write:** `BufferedOutputStream` wrapping `Files.newOutputStream`
- **Buffer:** configurable, default 64 KB
- **NFS:** kernel-transparent — mounts appear as POSIX paths
- **Why not FileChannel.transferTo?** Requires both endpoints to be file-backed.
  Our orchestrator pipes across heterogeneous backends (S3→NFS, URL→S3).

### S3Provider
- **Read:** `S3Client.getObject()` → `ResponseInputStream` (HTTP chunked, lazy)
- **Write:** Custom `S3MultipartOutputStream`:
  1. `createMultipartUpload` → gets uploadId
  2. Bytes fill an 8 MB part buffer
  3. Buffer full → `uploadPart` sends to S3, buffer reused (no new alloc)
  4. `close()` → flush final part + `completeMultipartUpload`
  5. Exception → `abortMultipartUpload` (cleanup orphaned parts)
- **MinIO:** `forcePathStyle(true)` + endpoint override

### UrlProvider
- **Read:** `HttpClient.send()` + `BodyHandlers.ofInputStream()` (lazy stream)
- **Write:** PipedOutputStream/PipedInputStream architecture:
  ```
  Orchestrator.write() → PipedOutputStream → PipedInputStream → HTTP request body
  ```
  The `HttpClient.sendAsync()` reads from the pipe on a virtual thread while
  the orchestrator writes into it. On `close()`, the pipe closes, HTTP completes,
  and the response status is checked.
- **Methods:** PUT (default) or POST, configurable via `--http-method`

## Memory Profile

| Transfer Path     | Read Buffer | Copy Buffer | Write Buffer   | **Total Peak** |
|-------------------|-------------|-------------|----------------|----------------|
| Local → Local     | 64 KB       | 64 KB       | 64 KB          | **192 KB**     |
| Local → S3        | 64 KB       | 64 KB       | 8 MB (1 part)  | **~8.19 MB**   |
| S3 → Local        | 64 KB       | 64 KB       | 64 KB          | **192 KB**     |
| S3 → S3           | 64 KB       | 64 KB       | 8 MB           | **~8.19 MB**   |
| URL → S3          | 64 KB       | 64 KB       | 8 MB           | **~8.19 MB**   |
| S3 → URL          | 64 KB       | 64 KB       | 64 KB (pipe)   | **192 KB**     |
| Local → URL       | 64 KB       | 64 KB       | 64 KB (pipe)   | **192 KB**     |
| NFS → S3          | 64 KB       | 64 KB       | 8 MB           | **~8.19 MB**   |

**Key insight:** The worst case is any path writing to S3 (8 MB multipart buffer).
All other paths stay under 200 KB regardless of file size. A 10 GB file uses the
same memory as a 10 KB file.

## CLI Usage

```bash
java -jar file-shuttle.jar \
  --file <logical-name> \
  --input-type <local|s3|nfs|url> \
  --input <source-address> \
  --output-type <s3|nfs|url|local> \
  --output <dest-address> \
  [--buffer-size 65536] \
  [--multipart-size 8388608] \
  [--http-method PUT|POST] \
  [--s3-endpoint http://minio:9000]
```

### Examples

```bash
# Local file → S3
file-shuttle -f report.csv -it local -i /data/report.csv \
  -ot s3 -o s3://my-bucket/reports/report.csv

# S3 → NFS share
file-shuttle -f backup.tar.gz -it s3 -i s3://backups/daily.tar.gz \
  -ot nfs -o /mnt/nfs-share/backups/daily.tar.gz

# URL download → S3
file-shuttle -f dataset.zip -it url -i https://example.com/data.zip \
  -ot s3 -o s3://datasets/incoming/data.zip

# S3 → HTTP POST upload
file-shuttle -f export.json -it s3 -i s3://exports/data.json \
  -ot url -o https://partner-api.example.com/upload --http-method POST
```

## Docker Compose Quick Start

```bash
# 1. Generate test data + run all 7 scenarios
chmod +x run-scenarios.sh
./run-scenarios.sh

# 2. Or run individual transfers
docker compose up -d minio nfs-server upload-server
docker compose run --rm file-shuttle \
  -f myfile.bin -it local -i /data/local/test-file.bin \
  -ot s3 -o s3://test-bucket/myfile.bin \
  --s3-endpoint http://minio:9000

# 3. Check results
# MinIO Console: http://localhost:9001 (minioadmin/minioadmin)
# Upload Server: curl http://localhost:8080/test.json

# 4. Tear down
docker compose down -v
```

### Services

| Service        | Port  | Purpose                            |
|----------------|-------|------------------------------------|
| minio          | 9000  | S3-compatible API                  |
| minio          | 9001  | Web console                        |
| nfs-server     | 2049  | NFS v4 export                      |
| upload-server  | 8080  | Mock HTTP upload/download API      |
| file-shuttle   | —     | One-shot CLI (exits after transfer)|
